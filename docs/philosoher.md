# Philosophers

このプロジェクトでは、プロセスにおけるスレッド処理の基礎を学ぶ。  
スレッドの作成方法と、`mutex` の使い方を学習する。

## Chapter I

### はじめに

哲学（ギリシャ語の `philosophia` に由来し、文字通りには「知を愛すること」）とは、存在、知識、価値、理性、精神、言語に関する一般的かつ根本的な問いを探究する学問である。こうした問いは、分析または解決すべき問題として定式化されることが多い。この語はおそらくピタゴラス（紀元前570年頃から紀元前495年頃）によって作られた。哲学的方法には、問いかけ、批判的議論、合理的論証、体系的な提示が含まれる。

古典的な哲学上の問いには、次のようなものがある。何かを本当に知り、それを証明することはできるのか。最も実在的なものは何か。哲学者たちは、より実践的で具体的な問いも立てる。最善の生き方は存在するのか。もし罰を受けずに済むなら、正しくあることと不正であることのどちらが良いのか。人間には自由意志があるのか。

歴史的には、「哲学」という語はあらゆる知の体系を指していた。古代ギリシャの哲学者アリストテレスの時代から19世紀に至るまで、「自然哲学」は天文学、医学、物理学を含んでいた。たとえばニュートンの1687年の著作 *Mathematical Principles of Natural Philosophy* は、のちに物理学の書物として分類された。

19世紀には、近代的な研究大学の発展により、学術哲学およびその他の分野は専門化・細分化されていった。現代では、従来哲学の一部であった探究のいくつかが、心理学、社会学、言語学、経済学などの独立した学問分野となっている。

一方で、芸術、科学、政治、その他の営みと密接に関わる探究の一部は、今なお哲学の一部にとどまっている。たとえば、美は客観的なものか主観的なものか。科学的方法は多数存在するのか、それとも一つだけなのか。政治的ユートピアは希望に満ちた夢なのか、それとも絶望的な幻想なのか。

学術哲学の主要な下位分野には、形而上学（「現実と存在の根本的な性質」を扱うもの）、認識論（「知識の性質と根拠、そしてその限界と妥当性」を扱うもの）、倫理学、美学、政治哲学、論理学、科学哲学が含まれる。

## Chapter II

### 共通指示

- プロジェクトは C で書かなければならない。
- プロジェクトは Norm に従って書かなければならない。ボーナス用のファイルや関数がある場合も、それらは Norm チェックの対象に含まれ、Norm エラーがあれば `0` 点になる。
- 未定義動作を除き、関数が予期せず終了してはならない（セグメンテーションフォルト、バスエラー、二重 `free` など）。そのようなことが起きた場合、プロジェクトは機能していないと見なされ、評価では `0` 点になる。
- ヒープに確保したメモリは、必要に応じてすべて適切に解放しなければならない。メモリリークは許容されない。
- 課題で要求されている場合は、ソースファイルを必要な出力物へコンパイルする `Makefile` を提出しなければならない。コンパイルには `cc` を用い、フラグ `-Wall`、`-Wextra`、`-Werror` を付けること。さらに、`Makefile` は不要な再リンクを行ってはならない。
- `Makefile` には最低限 `$(NAME)`、`all`、`clean`、`fclean`、`re` のルールを含めなければならない。
- ボーナスを提出する場合は、`Makefile` に `bonus` ルールを含めなければならない。このルールは、メインパートでは許可されていない各種ヘッダー、ライブラリ、関数を追加するものとする。ボーナスは、課題文で別指定がない限り、`_bonus.{c/h}` ファイルに配置しなければならない。Mandatory part と Bonus part の評価は別々に行われる。
- `libft` の使用が許可されているプロジェクトでは、そのソースと対応する `Makefile` を `libft` フォルダへコピーしなければならない。プロジェクトの `Makefile` は、まず `libft` の `Makefile` を使ってライブラリをコンパイルし、その後にプロジェクト本体をコンパイルしなければならない。
- プロジェクトには提出不要かつ採点対象外ではあるものの、テストプログラムを作ることを推奨する。これにより、自分の成果物や他人の成果物を簡単に試せるようになる。これらのテストは defense の際に特に有用である。実際、defense 中は、自分のテストおよび／または評価相手のテストを自由に使ってよい。
- 割り当てられた Git リポジトリに成果物を提出すること。採点対象になるのは Git リポジトリ内の成果物のみである。Deepthought が採点を担当する場合、それはピア評価の後に行われる。Deepthought による採点中に、成果物のいずれかの部分でエラーが発生した場合、その時点で評価は停止する。

## Chapter IV

### 概要

この課題で成功するために知っておくべき重要事項は以下のとおりである。

- 1人以上の哲学者が円卓に座っている。
- テーブルの中央には大きなスパゲッティのボウルが置かれている。
- 哲学者は「食べる」「考える」「眠る」を順番に繰り返す。
- 食べている間、哲学者は考えず、眠らない。
- 考えている間、哲学者は食べず、眠らない。
- 当然ながら、眠っている間、哲学者は食べず、考えない。
- テーブルの上にはフォークも置かれており、その本数は哲学者の人数と同じである。
- スパゲッティをフォーク1本だけで食べるのは現実的ではないため、哲学者は食事を始める前に、右手側と左手側の両方のフォークを取らなければならない。
- 哲学者は食事を終えると、フォークをテーブルに戻して眠り始める。目を覚ますと、再び考え始める。シミュレーションは、哲学者が1人餓死した時点で停止する。
- すべての哲学者は食事を必要とし、決して飢えてはならない。
- 哲学者同士は互いに通信しない。
- 哲学者は、他の哲学者が死にかけているかどうかを知らない。
- 言うまでもなく、哲学者は死を避けるべきである。

## Chapter V

### 全体ルール

Mandatory part 用のプログラムと、Bonus part 用の別プログラム（ボーナスを実施する場合）を書く必要がある。両方とも次のルールに従わなければならない。

- グローバル変数は禁止。
- プログラムは次の引数を受け取らなければならない。

```text
number_of_philosophers time_to_die time_to_eat time_to_sleep
[number_of_times_each_philosopher_must_eat]
```

- `number_of_philosophers`: 哲学者の人数であり、同時にフォークの本数でもある。
- `time_to_die`（ミリ秒）: 哲学者が最後の食事開始時刻、またはシミュレーション開始時刻から `time_to_die` ミリ秒以内に食事を開始しなければ、その哲学者は死亡する。
- `time_to_eat`（ミリ秒）: 哲学者が食事に要する時間。この間、哲学者は2本のフォークを保持していなければならない。
- `time_to_sleep`（ミリ秒）: 哲学者が眠る時間。
- `number_of_times_each_philosopher_must_eat`（任意引数）: すべての哲学者が少なくともこの回数だけ食事を終えた場合、シミュレーションは停止する。指定されない場合は、哲学者が死亡したときにシミュレーションを停止する。
- 各哲学者には `1` から `number_of_philosophers` までの番号が割り当てられる。
- 哲学者 `1` は、哲学者 `number_of_philosophers` の隣に座る。その他の哲学者 `N` は、哲学者 `N - 1` と哲学者 `N + 1` の間に座る。

プログラムのログについて:

- 哲学者の状態変化は、必ず次の形式で出力しなければならない。

```text
timestamp_in_ms X has taken a fork
timestamp_in_ms X is eating
timestamp_in_ms X is sleeping
timestamp_in_ms X is thinking
timestamp_in_ms X died
```

- `timestamp_in_ms` は現在のタイムスタンプ（ミリ秒）に、`X` は哲学者番号に置き換える。
- 表示される状態メッセージは、他のメッセージと重なってはならない。
- 哲学者の死を通知するメッセージは、実際の死亡から 10 ms 以内に表示しなければならない。
- 繰り返しになるが、哲学者は死を避けるべきである。

プログラムには `data race` があってはならない。

- Description: スレッドと `mutex` による Philosophers

Mandatory part 固有のルールは以下のとおりである。

- 各哲学者は、それぞれ独立したスレッドとして表現しなければならない。
- 各哲学者の組の間には1本のフォークがある。したがって、哲学者が複数いる場合、各哲学者は左側に1本、右側に1本のフォークを持つ。哲学者が1人しかいない場合、その哲学者が利用できるフォークは1本だけである。
- 哲学者がフォークを重複して使用しないよう、各フォークの状態は `mutex` で保護しなければならない。

## Chapter VII

### README 要件

Git リポジトリのルートには `README.md` ファイルを置かなければならない。その目的は、このプロジェクトを知らない人（ピア、スタッフ、採用担当者など）が、プロジェクトの内容、実行方法、さらに詳しい情報をどこで見つけられるかを素早く理解できるようにすることである。

`README.md` には最低限、次を含めなければならない。

- 最初の1行はイタリック体で、次の内容でなければならない。  
  *This project has been created as part of the 42 curriculum by <login1>[, <login2>[, <login3>[...]]].*
- `Description` セクション。プロジェクトの目的と簡潔な概要を含め、内容を明確に説明すること。
- `Instructions` セクション。コンパイル、インストール、および／または実行に関する関連情報を含めること。
- `Resources` セクション。トピックに関連する一般的な参考資料（ドキュメント、記事、チュートリアルなど）を列挙し、あわせて AI をどの作業に、プロジェクトのどの部分で使ったのかを説明すること。

プロジェクトによっては、追加セクション（使用例、機能一覧、技術的選択など）が必要になる場合がある。必要な追加項目がある場合は、下に明示される。

README は英語で書かなければならない。

Bonus part 固有のルールは以下のとおりである。

- すべてのフォークはテーブル中央に置かれる。

Bonus part は、Mandatory part が **完璧** な場合にのみ評価される。ここで「完璧」とは、Mandatory part が完全に実装され、異常なく動作することを意味する。Mandatory part の要件を **すべて** 満たしていない場合、Bonus part は一切評価されない。

## Chapter IX

### 提出とピア評価

課題は通常どおり Git リポジトリに提出すること。defense で評価されるのは、リポジトリ内にある成果物だけである。ファイル名が正しいことは、自分でよく確認しておくこと。

- Mandatory part directory: `philo/`
- Bonus part directory: `philo_bonus/`

評価中には、プロジェクトに対する短い修正を求められることがある。たとえば、ちょっとした挙動変更、数行のコードの追加や書き換え、あるいは簡単に追加できる機能などである。

この手順はすべてのプロジェクトに適用されるとは限らないが、評価ガイドラインに記載されている場合には、対応できるよう準備しておかなければならない。

この手順の目的は、プロジェクトの特定部分について本当に理解しているかどうかを確認することである。修正は、自分が選ぶ任意の開発環境（たとえば普段使っている環境）で行ってよく、評価の一部として特定の制限時間が定められていない限り、数分以内で完了できる程度のものでなければならない。

たとえば、関数やスクリプトに小さな更新を加える、表示内容を変更する、新しい情報を保存するためにデータ構造を調整する、といったことを求められる可能性がある。

詳細（範囲、対象など）は評価ガイドラインで指定され、同じプロジェクトであっても評価ごとに異なる場合がある。
